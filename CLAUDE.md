## Architecture

### Authentication Flow
- Better Auth handles user registration/login on the **frontend** (Next.js API routes).
- The frontend sends a **session token** (Bearer) with every API request to the backend.
- The backend (`middleware/auth.py`) validates the token by querying the shared **`session` table** in Neon PostgreSQL — no JWT signing, pure DB lookup.
- All task routes verify `url_user_id == session.userId` to enforce data isolation.

### API Structure
All task endpoints follow the pattern: `/api/{user_id}/tasks[/{task_id}]`

| Method | Path | Description |
|---|---|---|
| GET | `/health` | Health check |
| GET | `/api/{user_id}/tasks` | List tasks (filterable by `?completed=`) |
| POST | `/api/{user_id}/tasks` | Create task |
| GET | `/api/{user_id}/tasks/{task_id}` | Get single task |
| PUT | `/api/{user_id}/tasks/{task_id}` | Update task |
| DELETE | `/api/{user_id}/tasks/{task_id}` | Delete task |
| PATCH | `/api/{user_id}/tasks/{task_id}/complete` | Toggle completion |
| PATCH | `/api/{user_id}/tasks/{task_id}/pin` | Toggle pin |

### Task Model Fields
`id`, `user_id`, `title` (max 200), `description` (max 1000), `completed`, `due_date`, `priority` (high/medium/low), `is_pinned`, `created_at`, `updated_at`

**Default sort order:** pinned first → priority (high > medium > low) → newest first.

## Spec-Driven Development Workflow

**The Constitution governs everything** — see `.specify/memory/constitution.md`.

1. **Specify (WHAT):** Write requirements in `specs/<feature>/spec.md`
2. **Plan (HOW):** Design architecture in `specs/<feature>/plan.md`
3. **Tasks (BREAKDOWN):** Atomic testable units in `specs/<feature>/tasks.md`
4. **Implement (CODE):** Claude Code generates code from tasks — no manual coding

**Rule:** No code without a task. No task without a spec. Specs live in `specs/`; they are the single source of truth.

### PHR (Prompt History Records)
After each session, a PHR must be saved to `history/prompts/`. See `GEMINI.md` for the full PHR protocol. PHRs are routed by stage: `constitution/`, `<feature-name>/`, or `general/`.

### ADR (Architecture Decision Records)
Significant architectural decisions go in `history/adr/`. Never auto-create — always get user consent first.

## Key Constraints & Rules

- **No manual coding** — all code must be AI-generated from refined specs.
- **No hardcoded secrets** — use `.env` files; never commit them.
- **Data isolation is non-negotiable** — every DB query filters by `user_id`.
- **Smallest viable diff** — do not refactor unrelated code in any PR.
- **CORS** is locked to `http://localhost:3000` in dev — update `main.py` for production.
- **`BETTER_AUTH_SECRET` must match** between frontend and backend — it's the shared session signing secret.
- The `.specify/memory/constitution.md` is the authoritative reference; it supersedes everything else in case of conflict.
